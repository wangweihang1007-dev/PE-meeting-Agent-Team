# 投资项目表录入

## 使用边界

这个 Skill 只负责“项目录入表”环节，不负责生成会议纪要本身。

在完整自动化流程中：

1. 会议纪要流水线先运行：会议原文 + BP/公司材料 → 修正 → QA → 总结 → Word/MD 会议纪要。
2. 项目录入流水线后运行：最终会议纪要 + BP/公司材料 + 企查查/工商资料 + 用户补充资料 → 项目录入表。
3. 企查查/工商资料只用于项目录入，不得进入会议纪要的修正、QA、总结环节，也不得反向改写会议纪要。

## 开始前

1. 读取 `references/field-schema.md`，确认新版 A:Q 列、口径和留空规则。
2. 读取 `references/verification-and-style.md`，执行核查、冲突处理、白描和低 bit 写法。
3. 对齐格式时，优先检查：
   - `assets/项目表-录入参考指引.xlsx`
   - `assets/硕成_项目录入表.xlsx`
4. 处理 PDF、DOCX、PPTX、XLSX、TXT、MD 时，可运行：

```powershell
python scripts/extract_reference_text.py <材料路径> --output-dir <临时目录>
```

5. 新版项目表有效列为 A:Q；P 为备注，Q 为否决原因；旧版 `是否通过` 字段已取消。

## 工作流

### 1. 盘点资料

- 按文件名、日期、类型列出全部材料。
- 标记资料类型：最终会议纪要、会议逐字稿、BP/路演材料、企查查/工商、财报/审计、补充说明。
- 识别资料时点；旧 BP 不得自动覆盖较新的会议、工商或融资信息。
- 先提取原文，再归纳。保留文件名、页码/段落或材料类型作为内部证据定位。
- 文件名日期与正文日期冲突时，以正文可验证日期为准并记录异常。

### 2. 建立事实台账

在临时目录生成 `source_facts.json`，每条事实至少包含：

```json
{
  "field": "本轮估值",
  "value": "7.4亿元",
  "status": "confirmed",
  "source": "20260609_会议纪要.docx",
  "source_type": "纪要",
  "locator": "融资",
  "as_of": "2026-06-09",
  "notes": ""
}
```

`status` 只使用：

- `confirmed`：两个独立来源一致，或一个高权威来源明确。
- `company_claim`：仅来自 BP、管理层或会议陈述。
- `inferred`：根据明确事实做的有限归纳；不得填入精确数字。
- `conflict`：不同来源冲突，尚不能消解。
- `missing`：无可靠信息。

### 3. 核查关键事实

必须核查：

- 公司全称、成立时间、注册城市、实控人和最新股权。
- 历史融资轮次、时间、金额、投资方、投前/投后估值。
- 本轮估值、融资额、截止时间是否为当前口径。
- 主要产品的量产、送样、在研状态。
- 客户是“已供货/量产”“送样验证”“接触中”中的哪一种。
- 收入、利润是确认收入、发货、订单、预计还是预算。
- 市场规模的口径、年份、区域、来源是否一致。

优先使用用户提供的企查查/天眼查/爱企查摘录或其他工商材料。用户未提供、材料过旧或不同来源冲突时，不得自行补猜；留空或列为待核查。仅在用户明确要求联网搜索时再查询外部工商、融资或市场信息。

### 4. 处理冲突

- 冲突判断顺序：工商登记 > 官方披露/公告/审计报表 > 口头陈述/BP。
- 同类来源中，较新时点优先。
- 不设静默的全局来源优先级；每个关键数字和状态都要能说明口径来自哪个来源。
- 经营现状优先使用较新的会议纪要或逐字稿；BP 用于补充产品、技术和历史。
- 公司目标、预测和管理层口径必须写“计划”“预计”“公司表示”。
- 无法消解的冲突不得删掉异口径：表格用保守表述或留空；备注或交付说明保留冲突话语和来源。
- 不把发货额写成收入，不把意向客户写成客户，不把送样写成量产，不把规划产能写成现有产能。

### 5. 起草录入行

- 严格按 `references/field-schema.md` 的新版 A:Q 列顺序。
- “价值”栏固定使用七段：`1.团队`、`2.股权结构`、`3.产品`、`4.技术`、`5.生产、客户`、`6.市场`、`7.收入`。
- 主营业务写产品级短语，不写“半导体材料”“元器件”“设备”等宽泛类别。
- 使用白描：去掉形容词、副词和宣传口号；优先用数字、年份、轮次、状态、客户阶段。
- 能用缩略语就不用全称：BP、纪要、摘要、QCC、财报、工商、官网、公告等。
- 历史实际与未来预测分开写，保留年份和单位。
- `否决原因` 不得代替投资人判断，除非用户明确提供结论。
- P 列 `备注` 必填，用于标明资料来源组合和未消解冲突；如 `资料来源：BP、纪要、QCC`。

### 6. 自动校验

先把草稿保存为 JSON，再运行：

```powershell
python scripts/validate_entry.py draft.json
```

修复 `error`；逐项审阅 `warning`。校验不代替事实核查。

### 7. 写入 Excel

- 优先复制用户提供的项目表；未提供时复制 `assets/项目表-录入参考指引.xlsx`。
- 新版项目表使用 A:Q；不得生成旧版 `是否通过` 列。
- 在现有数据末行追加，不覆盖说明行或其他项目。
- 复制相邻正式项目行的格式，仅写入值。
- 日期写真实 Excel 日期；金额和百分比保持与少样本一致的中文单位表达。
- 可运行：

```powershell
python scripts/write_entry_excel.py draft.json --workbook assets/项目表-录入参考指引.xlsx --output <输出路径.xlsx>
```

- 完成后检查 A:Q 的值、格式和换行。

## 完成交付

交付：

- 一份最终 `.xlsx`。
- 一段不超过 6 条的核查说明，只列冲突、缺失和重要口径限制。

不要交付抽取文本、事实台账、脚本输出或预览图，除非用户要求。
